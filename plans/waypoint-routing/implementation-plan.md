# Waypoint Routing Implementation Plan

## Phase 1: Point-Based UI On CycleWays Network

Status: implemented.

### Scope

Keep all routing on the existing CycleWays graph, but make the public UI present
waypoints as the primary route-building primitive.

### Tasks

- [x] Audit current user-facing copy that says "selected segments" or implies manual
  segment selection.
- [x] Change the empty state to instruct users to click map points.
- [x] Treat segment click behavior as "add route waypoint near this segment" instead
  of "select this segment".
- [x] Keep hover/focus segment behavior for inspection.
- [x] Ensure point markers are the visible editable route handles.
- [x] Replace selected segment highlighting with clipped waypoint route geometry.
- [x] Keep remove/drag waypoint behavior working.
- [x] Add temporary waypoint URL parsing and serialization for local Phase 1
  testing.
- [x] Preserve existing `route=` segment-ID links for backwards compatibility.
- [x] Add URL length guardrails.
- [x] Add analytics naming that distinguishes waypoint route creation from legacy
  segment selection.
- [x] Add tests around snap rejection, clipped route geometry, and waypoint URL
  parsing.
- [x] Scope route warnings/data points to the clipped route geometry.
- [x] Keep segment hover/focus showing all segment data for inspection.
- [x] Add tests around route data point location filtering.
- [ ] Consider waypoint reordering controls if route editing needs it later.

### Acceptance Criteria

- A user can build a route by clicking waypoint locations without understanding
  segment names.
- The first point creates a marker only; the route appears after the second
  point.
- The visible route starts at the first snapped waypoint and ends at the last
  snapped waypoint.
- Same-segment routes render only the slice between the two waypoints.
- GPX/elevation/distance use clipped route geometry, not full selected segments.
- Existing route links using `route=` still load.
- New public route links use `route=` with a versioned compact payload, not
  decimal waypoint URLs.
- Route warnings/data points trigger only when the clipped route geometry passes
  near their GPS point, with segment-wide fallback for entries that do not have
  a location.

### Notes

- Phase 1 rejects off-network points with a message. Phase 2 will replace this
  rejection with external routing for off-network legs.
- The branch-local `w=` format was replaced with versioned `route=` payloads
  before promotion.
- The summary modal still lists the underlying route segments as route details,
  but the primary route state is the user's route points.
- Selected segments are still computed as metadata, but they no longer drive the
  visible route line.

## Compact Route Encoding

Status: implemented.

### Scope

Replace the temporary decimal waypoint URL with a compact Base58 binary route
payload that keeps the version inside the payload and can survive future segment
splits better than `segmentId + offset`.

### Tasks

- [x] Add route payload version 3 to `utils/route-encoding.js`.
- [x] Keep version 1/2 `route=` decoding for existing segment-ID URLs.
- [x] Remove `w=` generation and decoding before this branch is promoted.
- [x] Add binary helpers for:
  - varint encode/decode
  - zig-zag signed integer encode/decode
  - quantized `[lng, lat]` coordinate encode/decode
- [x] Add a route tokenizer that emits:
  - quantized coordinate anchors for clipped starts/ends and mid-route user
    points
  - full segment ID hint tokens for full CycleWays segments between anchors
  - reserved external coordinate anchors for Phase 2
- [x] Decode version 3 by restoring route points and using segment IDs as hints where
  they still exist.
- [x] Add tests for:
  - round-trip route payloads
  - backwards-compatible legacy `route=` decoding
  - URL length compared with current decimal waypoint URLs
  - coordinate-anchor based restoration
- [x] Update sharing analytics to report payload version and token counts, not raw
  coordinates.

### Acceptance Criteria

- New shares use only `route=<base58>`.
- The payload includes its version byte.
- Existing production `route=` links still load.
- `w=` is not generated by the share flow.
- Shared routes with partial first/last segments reopen at the same visible
  start/end points within the chosen coordinate quantization tolerance.
- A route that includes a segment later split can still be reconstructed from
  coordinate anchors and current routing data.

## Phase 2: External Routing Fallback

Status: planned.

### Scope

When a waypoint cannot snap to the CycleWays graph, use an external OSM routing
engine for that leg.

### Tasks

- Define the CycleWays snap threshold and surface it in route diagnostics.
- Add route leg model with `cycleways` and `external` leg types.
- Add provider interface:

```js
async function routeExternalLeg(start, end, options) {
  return {
    provider: "graphhopper",
    profile: "bike",
    distanceMeters: 0,
    coordinates: []
  };
}
```

- Start with a provider adapter for GraphHopper or a locally configured routing
  endpoint.
- Render external legs as dashed gray route lines.
- Include external coordinates in GPX export.
- Exclude external legs from CycleWays warning/data/quality summaries.
- Add clear UI messaging when a route uses non-CycleWays connectors.
- Add fallback/error handling when the external provider is unavailable.
- Decide whether mixed routes can be shared with compact coordinate anchors only,
  or require a backend snapshot later.

### Acceptance Criteria

- Points near CycleWays still prefer CycleWays.
- Points outside the threshold can still produce a route through an external
  routing engine.
- External legs are visually distinct.
- GPX export includes both CycleWays and external legs in order.
- If external routing fails, the user gets a clear, recoverable error.

## Suggested Sequencing

1. Implement Phase 1 point placement and clipped route geometry.
2. Replace temporary `w=` links with versioned compact `route=` links.
3. Add tests around snap rejection, clipped geometry, and compact route parsing.
4. Add the explicit route leg model behind the scenes.
5. Add one external provider behind a feature flag.
6. Decide on shareability limits for mixed routes after testing URL sizes.
